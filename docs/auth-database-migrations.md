# Auth & invite database migrations

How to generate and apply Prisma migrations for **[Better Auth](https://www.better-auth.com)** (Discord OAuth) plus the custom **`InviteToken`** table.

Better Auth is configured in [`apps/api/src/lib/auth.ts`](../apps/api/src/lib/auth.ts). The Prisma schema lives in [`packages/database/prisma/schema.prisma`](../packages/database/prisma/schema.prisma).

---

## Why two steps?

| Tool                       | Works with Prisma? | Role                                                                                                  |
| -------------------------- | ------------------ | ----------------------------------------------------------------------------------------------------- |
| **`better-auth generate`** | Yes                | Writes `User`, `Session`, `Account`, `Verification` models into `schema.prisma` from your auth config |
| **`better-auth migrate`**  | **No**             | Only for the built-in Kysely adapter ‚Äî not for `@better-auth/prisma-adapter`                          |
| **`prisma migrate dev`**   | Yes                | Creates SQL migration files and applies them to PostgreSQL                                            |

With the Prisma adapter you always: **generate schema ‚Üí Prisma migrate**.

---

## Prerequisites

1. **`DATABASE_URL`** in [`packages/database/.env`](../packages/database/.env) (see [`.env.example`](../packages/database/.env.example)).
2. PostgreSQL running and reachable (`pnpm db:status` from the repo root).
3. Workspace dependency **`@better-auth/cli`** in `apps/api` (already installed).

---

## Flow (first time or after auth config changes)

### 1. Add custom models first

Put app-specific tables (e.g. **`InviteToken`**) in `schema.prisma` **before** running the Better Auth CLI. The generator merges into the same file; it does not invent custom tables.

Example (already in this repo):

```prisma
model InviteToken {
  id        String    @id @default(cuid())
  token     String    @unique
  used      Boolean   @default(false)
  usedBy    String?
  usedAt    DateTime?
  expiresAt DateTime
  createdAt DateTime  @default(now())
}
```

### 2. Generate Better Auth Prisma models

From the **repository root** (PowerShell):

```powershell
cd apps/api
pnpm exec better-auth generate `
  --config ./src/lib/auth.ts `
  --output ../../packages/database/prisma/schema.prisma `
  -y
```

Bash (same paths):

```bash
cd apps/api
pnpm exec better-auth generate \
  --config ./src/lib/auth.ts \
  --output ../../packages/database/prisma/schema.prisma \
  -y
```

**Flags:**

| Flag       | Meaning                                                               |
| ---------- | --------------------------------------------------------------------- |
| `--config` | Auth entry point (must export `auth` from `betterAuth(...)`)          |
| `--output` | Target `schema.prisma` (monorepo database package)                    |
| `-y`       | Overwrite without prompts ‚Äî **review the git diff** before committing |

This adds (or updates) **`User`**, **`Session`**, **`Account`**, and **`Verification`**, including `@@map(...)` table names expected by Better Auth.

**Warning:** `-y` replaces the entire output file. Keep game models (`Character`, `CharacterSheet`) and `InviteToken` in the file before generating, or restore them from git if the CLI drops them.

### 3. Regenerate the Prisma client

From the **repository root**:

```bash
pnpm db:generate
```

### 4. Create and apply the migration (development)

From the **repository root**:

```bash
pnpm db:migrate
```

Prisma prompts for a migration name (e.g. `add_better_auth_and_invite`).

Or, with an explicit name, from `packages/database`:

```bash
cd packages/database
pnpm exec prisma migrate dev --name add_better_auth_and_invite
```

### 5. Build the database package

```bash
pnpm build:database
```

Required before `apps/api` or other packages that import `@galipette/database`.

---

## Deploy (CI / production)

Apply existing migration SQL only ‚Äî do not run `better-auth generate` on production servers:

```bash
pnpm db:deploy
```

Typical order: `pnpm install` ‚Üí set **`DATABASE_URL`** ‚Üí **`pnpm db:deploy`** ‚Üí start API.

---

## Checklist after schema changes

1. `better-auth generate` (when auth plugins/providers change)
2. `pnpm db:generate`
3. `pnpm db:migrate` (dev) or `pnpm db:deploy` (prod)
4. `pnpm build:database`
5. Commit **`schema.prisma`** and **`prisma/migrations/**/migration.sql`\*\*

## Troubleshooting

| Issue                                     | What to do                                                                           |
| ----------------------------------------- | ------------------------------------------------------------------------------------ |
| `better-auth migrate` does nothing useful | Expected with Prisma ‚Äî use **`generate` + `prisma migrate`** only                    |
| CLI asks to overwrite schema              | Pass **`-y`** or answer yes; then review diff                                        |
| Auth tables missing at runtime            | Run **`pnpm db:migrate`** or **`pnpm db:deploy`**; confirm **`pnpm build:database`** |
| CLI / runtime version skew                | Align `@better-auth/cli` with `better-auth` in `apps/api/package.json`               |
| DB unreachable                            | Fix **`DATABASE_URL`**, then **`pnpm db:status`**                                    |
